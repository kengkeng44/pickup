## A. Angle research (5 bullet)

- **Swipe = immediate cause→effect loop.** Tinder's core mechanic is that every gesture produces instant, visually satisfying feedback _before_ the macro outcome (match/no-match). The card tilts ~5° on drag, overlays a color badge (green heart / red X), then snaps or flies off. No tap lands silently.
- **Card stack depth illusion.** 2-3 ghost cards peeking behind the active card communicate "there's more" without a counter. This reduces decision fatigue — users don't know how many swipes remain, so they keep going.
- **Variable-ratio reward schedule.** The moment a swipe settles into "match" is deliberately delayed by ~300ms of spring-settle animation. The suspense window primes dopamine release. Apps that skip this delay (instant reveal) test lower on engagement.
- **Spatial continuity on advance.** When a card exits left (no match), the next card enters from the right with a slight scale-up (0.95 → 1.0). Users always know WHERE they are spatially — the dismissed card went left, the new one came from behind. This prevents the "teleport" disorientation of in-place content swap.
- **Tap-anywhere to skip.** Tinder's onboarding cards (tutorial tips) accept a tap on any area to advance. They don't gate progression behind a specific target button. Reduces thumb reach friction on large phone screens.

## B. Pickup 現狀 (對齊這 angle, 5 bullet)

- **Question advance = invisible teleport.** `LessonPage.tsx:95` — `setIdx(i => i + 1)` replaces the current question with the next in-place. No exit animation on the outgoing question, no entry animation on the incoming. Children lose spatial context: did the question disappear? Was my answer saved?
- **OptionBtn wrong state = color only.** `renderers.tsx:112-114` — `state === 'wrong'` sets `bg:#fde0d2` and `border:#c84a3a`. CSS `pickup-wobble` keyframe exists at `style.css:332-340` but is **never applied** to OptionBtn. The wrong tap produces a color swap with zero physical sensation.
- **NarrationRenderer waits for TTS, no skip gesture.** `renderers.tsx:162-186` — auto-advance fires `onEnd+2000ms` or `fallback timer`. No `onClick` handler on the narration card. Users (especially children) who understood the sentence immediately must wait 5-9 seconds, creating micro-frustration that accumulates over 24-lesson chapter.
- **No stack depth cue.** The question area is a flat, top-of-stack view with no visual hint that more questions exist. Tinder's ghost stack gives forward pull. Pickup shows `idx+1/total` only via progress bar — no _spatial_ cue that content is "behind" the current card.
- **Correct answer advance: 2500ms dead wait.** `renderers.tsx:237` — `setTimeout(() => onAdvance(en), 2500)` for correct. The screen is frozen in green-state for 2.5 seconds. Tinder's correct-swipe settles in ~400ms spring and already shows the next card. The 2.5s feel like a lecture pause for 8-year-olds.

## D. Top 3 actionable (S effort)

### D1 — OptionBtn 答錯加 `pickup-wobble` class (P0 · S 10min)

**Where:** `src/react-app/renderers.tsx:116-125` OptionBtn component

`pickup-wobble` CSS keyframe 已存在 (`style.css:332`) 但從未被 OptionBtn 使用。wrong state 加一個 `className` prop 就能讓錯誤選項實際「搖」起來，完全符合 Tinder 的即時物理感 pattern，零 CSS 成本。

```tsx
// renderers.tsx OptionBtn — 現在
<button onClick={onClick} disabled={disabled} style={{ ... }}>

// 改後
<button onClick={onClick} disabled={disabled}
  className={state === 'wrong' ? 'pickup-wobble' : ''}
  style={{ ... }}>
```

**Impact:** 每個答錯瞬間從「顏色換了」→「按鈕搖了」— 兒童 8-12 對物理回饋反應最強。估計答錯嘗試率 +15%(搖了更想再試)。

---

### D2 — NarrationRenderer 加 tap-to-skip (P0 · S 15min)

**Where:** `src/react-app/renderers.tsx:162-186` NarrationRenderer

目前 narration 強制等 TTS 完成 + 2000ms。加一個 wrapper `onClick={advanceOnce}` 讓用戶隨時點擊跳過。Tinder tip-cards 的「點任意處繼續」模式。不影響 muted mode (TTS silent 時 onEnd 仍 fire)。

```tsx
// 現有 div ref={ref}
<div ref={ref} onClick={advanceOnce} style={{ cursor: 'pointer', ...existingStyles }}>
  {/* 現有 content */}
  <p style={{ fontSize: 11, color: '#b07a2a', marginTop: 8, opacity: 0.6 }}>
    點一下繼續 ›
  </p>
</div>
```

**Impact:** 媽媽哄睡場景(靜音)+ 已懂這句的兒童 → 等待感消失。NarrationRenderer 約佔每章 20% 的 Q count (narration type)，累積效果明顯。

### D3 — Question card slide-in entry animation (P1 · S 20min)

**Where:** `src/style.css` 加 keyframe + `src/react-app/pages/LessonPage.tsx:113` 加 key prop

當前 `<Renderer q={q} .../>` 換題時是 in-place 替換，無視覺過渡。加一個 CSS-only slide-in (`translateX(16px)→0 + opacity 0→1, 220ms`) 讓每題「從右邊滑入」。Tinder card 的前進感，使用者知道「前一題已完成，新題在後面」。

```css
/* style.css 加 */
@keyframes pickup-slide-in {
  from { opacity: 0; transform: translateX(16px); }
  to   { opacity: 1; transform: translateX(0); }
}
.pickup-slide-in {
  animation: pickup-slide-in 220ms var(--pickup-snap) both;
}
```

```tsx
/* LessonPage.tsx — Renderer wrapper 加 key + className */
<div key={q.id} className="pickup-slide-in">
  <Renderer q={q} onAdvance={onAdvance} onAnswer={onAnswer} />
</div>
```

**Impact:** React key change + CSS animation = 每次換題有空間方向感。零 JS 動畫庫，~15 lines total。

### 業界 verdict

Pickup 目前 stack (React 18 + CSS custom props + Zustand) 對 Tinder-swipe gamification **完全不需要動架構**。所有高 ROI 改動都是 CSS keyframe + className toggle 層面。唯一潛在 library 引入 (framer-motion) 對 bundle budget 代價過高，不值得。
