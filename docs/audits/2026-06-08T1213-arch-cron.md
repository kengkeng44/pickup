## D. DRY violations — angle-specific findings

### D1. `loadChapterLessons` — 4 implementations
The canonical version in `src/data/lessons.ts:404` fetches, Zod-validates, caches via `Map<ChapterId, Lesson[]>`, and throws on error. Three divergent clones exist with different failure semantics:

| Location | Cache? | Validates? | On error |
|----------|--------|-----------|----------|
| `src/data/lessons.ts:404` (canonical) | ✅ Map | ✅ Zod | throws |
| `src/react-app/pages/MapPage.tsx:40` | ✅ plain object | ❌ | returns `[]` silently |
| `src/react-app/pages/ChapterIntroPage.tsx:33` | ❌ | ❌ | unhandled promise rejection |
| `src/react-app/components/KeySentencesSheet.tsx:32` | ❌ | ❌ | unhandled rejection |
| `src/react-app/pages/LessonPage.tsx:58` | ❌ | ❌ | unhandled rejection |

`MapPage.tsx` cache key is `number` not `ChapterId` — type mismatch vs canonical `Map<ChapterId, …>`. On a fetch failure for Ch7 inside `MapPage`, the lesson list silently returns `[]`, hiding the problem from the user and from PostHog.

### D2. `CHAPTER_META` — 4 out-of-sync definitions

| Location | Covers | Shape |
|----------|--------|-------|
| `src/data/storyKitten.ts:177` (canonical) | Ch1-7 | `{ titleZh, titleEn, accent, banner, … }` |
| `src/react-app/views/MapView.tsx:8` | Ch1-3 only | `{ id, titleZh, titleEn, unlocked, total }` |
| `src/react-app/pages/MapPage.tsx:86` | Ch1-31 | `{ titleZh, titleEn, accent }` |
| `src/react-app/pages/ChapterIntroPage.tsx:14` | **Ch1-8 only** | `{ zh, en, emoji }` |

**32 `lessons-ch*.json` files ship but ChapterIntroPage only covers Ch1-8.** Navigating to `/chapter/9/intro` → `CHAPTER_META[9] = undefined` → `meta?.zh` renders blank — user sees empty title and no emoji. MapPage correctly covers Ch1-31 but is not re-exported.

Field name fragmentation (`zh` / `titleZh` / `titleZh`) means no structural reuse is possible without a consolidation refactor.

### D3. `pickPraise()` — identical 1-liner copied twice

```ts
// src/ui/EndOverlay.ts:60-61  AND  src/ui/ClozeUI.ts:101-102
function pickPraise(pool: readonly string[]): string {
  return pool[Math.floor(Math.random() * pool.length)];
}
```

Identical implementation, private scope in both files. Belongs in `src/ui/domUtil.ts` (which already holds shared DOM helpers per v1.9.29 audit-8 pattern).

### D4. Coin localStorage key mismatch — **silent data loss**

`src/data/coins.ts` (canonical API): `LS_COINS = 'pickup.coins.total'`

`src/react-app/pages/MapPage.tsx:547`:
```ts
localStorage.setItem('pickup.coins', String(curCoins + 10));
```

When a user opens a treasure chest on the map: `readCoins()` is called (correctly reads `pickup.coins.total`), then the +10 is written to `pickup.coins` (different key). `readCoins()` on subsequent render reads `pickup.coins.total` unchanged — **chest reward is permanently lost**. The HUD coin display never increments. The commented-out `// addCoins(10)` in the same line was left as a dead comment when the author wrote the raw set as a placeholder, never wired up.

### D5. SRS queue key read outside canonical module

`src/scenes/StoryModeScene.ts:711` and `src/react-app/pages/LessonPage.tsx:462` both hard-code `localStorage.getItem('wordwar.srs.kitten')` — bypassing `src/data/storyKitten.ts:302` where `LS_SRS_QUEUE` is the single constant. If the key is ever changed, 2 callsites will silently read `null` instead of the queue. B.253 corrected the key mismatch in `LessonPage` (comment at line 460 documents the history) but the raw string still appears at line 462 rather than importing `LS_SRS_QUEUE`.

## E. Top 3 P0 architecture risks

### P0-1 — Treasure chest coins silently lost (`MapPage.tsx:547`) ⚠️

**File**: `src/react-app/pages/MapPage.tsx:547`
**Bug**: `localStorage.setItem('pickup.coins', ...)` writes to wrong key (`pickup.coins` ≠ `pickup.coins.total`). Every chest open increments a phantom key. `readCoins()` always reads 0 for this write.
**Fix** (5 min, 1 line): Replace with `addCoins(10)` — import already present at line 19.
```ts
// Before:
const curCoins = readCoins();
localStorage.setItem('pickup.coins', String(curCoins + 10));
// After:
addCoins(10);
```

### P0-2 — ChapterIntroPage blank titles for Ch9-31 ⚠️

**File**: `src/react-app/pages/ChapterIntroPage.tsx:14-24`
**Bug**: Local `CHAPTER_META` only covers Ch1-8. With 32 shipped chapters, any `/chapter/9/intro` through `/chapter/31/intro` deep-link renders `undefined` → empty `<h1>` and missing emoji. This is a visible regression for the 29 chapters shipped since B.249.
**Fix** (S, ~15 min): Import `CHAPTER_META` from `MapPage.tsx`'s definition (already covers Ch1-31) OR centralize into `src/data/chapterMeta.ts` and re-export. Quick path: merge ChapterIntroPage's local object into MapPage's (which has `accent` but not `emoji`) — add emoji field to MapPage's definition.

### P0-3 — `loadChapterLessons` clone in MapPage silently masks fetch failures ⚠️

**File**: `src/react-app/pages/MapPage.tsx:40-48`
**Risk**: `.catch(() => [])` swallows all network + parse errors and returns an empty array. A CDN hiccup on any chapter quietly shows 0 lessons to the user with no feedback. The canonical `src/data/lessons.ts` version throws, enabling error boundary display.
**Fix** (M, ~20 min): Delete local function, import `loadChapterLessons` from `src/data/lessons.ts` (already imported for types at line 20). Wrap callsites in try/catch to surface errors.
