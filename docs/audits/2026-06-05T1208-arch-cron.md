## D. Angle-specific risk: Zod v4 migration (5 bullets)

**D1. `z.preprocess()` removed in Zod v4** — `sentences.ts:87` uses `z.preprocess()` to stamp `type: 'listen-mc'` onto legacy JSON entries missing a `type` field. Zod v4 removes `z.preprocess`; replacement is `z.transform` with a `.pipe()` chain or `z.input` coercion. The existing vitest test `sentences-schema.test.ts > ClozeQuestionsSchema preprocess` directly exercises this path. Any `npm update zod` to v4 would break parse at runtime and fail 3 tests immediately.

**D2. `z.ZodIssueCode.custom` API change** — `lessons.ts:143` uses `z.ZodIssueCode.custom` inside a `superRefine` call to emit a cross-field validation error (correctOrder indices < tiles.length). In Zod v4, issue codes are restructured under `z.core` namespace. The string `"custom"` still works as a literal but the enum path `z.ZodIssueCode.custom` may not resolve. Low blast radius (only tap-tiles validation), but fails silently if the error code doesn't match.

**D3. `ChapterIdSchema` duplicated in two files** — Both `storyKitten.ts:21` and `lessons.ts:168` independently define the same `z.union([z.literal(0)..z.literal(7)])` with 8 literal arms. Zod v4 introduces `z.literal([0,1,2,3,4,5,6,7])` (array overload) and `z.union` with spread literal shorthand. Either way the current duplication means: adding a Ch8 requires updating both files. Historically this has already bitten the project (CLAUDE.md v2.0.B.204 mentions "Ch1 drop + renumber" — storyKitten.ts comment at line 288 notes the same renumber but independently). A single re-exported constant in lessons.ts with a re-export from storyKitten.ts would eliminate the drift surface.

**D4. `discriminatedUnion` + `superRefine` workaround becomes obsolete in v4** — `lessons.ts` applies `QuestionUnion.superRefine(...)` after the discriminated union specifically because "z.discriminatedUnion requires plain ZodObject members; ZodEffects from .refine() breaks it". In Zod v4 this restriction was lifted — discriminated unions accept ZodEffects members. The workaround code still works in v4, but the comment no longer describes reality. Future developers may try to simplify using the new pattern and introduce a subtle incompatibility if they half-migrate.

**D5. validate-lessons.js CI guard does NOT run the Zod schema** — The build-time guard (`tools/validate-lessons.js`) explicitly avoids importing the TypeScript Zod schemas (states "fast JSON shape checks"). Real Zod discriminated-union validation only happens via vitest (`tests/data/lessons-ch1-validate.test.ts` — covers ch0 + ch1 only). Ch2-Ch8 JSONs are validated only by mirror-lint rules, not by the full `LessonsSchema.parse()`. If a new question type (e.g. future `image-mc`) is added to the discriminated union but not to ch2-8 JSONs, it won't be caught until a user triggers a runtime parse error. A `tsx tools/validate-lessons.js` integration that imports and runs `LessonsSchema.parse()` would close this gap.

## E. Top 3 P0 architecture risks

**P0-1 — `z.preprocess` removal blocks semver-compliant upgrade** (lessons.ts / sentences.ts)
Zod publishes v4 as a new major. If a dependency (vite-plugin-xxx, vitest, etc.) pulls in zod@^4 via peer deps or the user runs `npm update`, the `ClozeQuestionsSchema` preprocess call breaks at runtime for any `sentences.json` entry missing a `type` field. The test suite catches this immediately (3 test failures), but production would also silently fail to load free-practice questions (sentences.json has no `type` field on legacy entries). Migration path: replace `z.preprocess` with `z.array(z.preprocess(...single item..., QuestionSchema))` or switch to input transforms. Estimated effort: 30 min.

**P0-2 — Ch2-Ch8 Zod validation gap in CI** (tools/validate-lessons.js + tests/)
`validate-lessons.js` runs at `npm run build` but only does structural JSON checks — it does NOT invoke `LessonsSchema.parse()`. The only real Zod test is `tests/data/lessons-ch1-validate.test.ts` which covers ch0+ch1 only. This means the 1100+ questions in ch2-ch8 JSONs (380 KB of lesson data) pass build with only mirror-lint checks. A schema-breaking question (wrong type discriminator, missing required field, 5-element options tuple) would reach production and cause a runtime `ZodError` crash in `loadChapterLessons()`. Fix: add `tests/data/lessons-ch2-8-validate.test.ts` mirroring the ch1 test, or extend validate-lessons.js to import via `tsx`. Estimated effort: 1 hr.

**P0-3 — Duplicate `ChapterIdSchema` creates future sync drift** (storyKitten.ts:21 + lessons.ts:168)
Both files export `ChapterIdSchema` as the same 8-literal union. Today they're identical. But `storyKitten.ts` is "Phaser legacy" and lessons.ts is "React-first active" — the next chapter addition or renumber (like B.204's Ch1 drop) requires updating both. CLAUDE.md history shows this exact drift already happened (v2.0.B.204 renumber comment appears independently in each file with slightly different wording). Simple fix: `storyKitten.ts` imports and re-exports `ChapterIdSchema` from `lessons.ts` instead of re-declaring. No runtime change, eliminates drift. Estimated effort: 5 min.
