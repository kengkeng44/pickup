## A. Anki SRS card design patterns (5 bullets)

- **Due count is the entry point.** Anki's home screen shows exactly how many cards are due *right now* in each deck — this single number drives daily habit. Users know before opening a deck whether they have work to do, which prevents the "is there anything to review?" ambiguity that causes churn.
- **4-button self-rating after reveal (Again / Hard / Good / Easy).** After seeing the answer, the user rates their own recall quality. Anki's SM-2 algorithm multiplies the card's interval by an ease factor derived from these ratings. This closes the feedback loop between user confidence and card scheduling, and gives users agency over their own curve.
- **Interval transparency on each rating button.** The due-in label (e.g. `<10m` / `1d` / `4d` / `12d`) appears on each rating button before the user taps. Users understand *when* the card will return — not as a black box. This builds trust and makes "Easy" feel like a real reward ("I won't see this for 12 days").
- **Leech detection.** Cards answered wrong 8+ times in Anki get automatically tagged "leech" and optionally auto-suspended. The UI flags them explicitly. Without this, perpetually hard cards silently drain user morale and inflate review time.
- **Cloze as typing recall, not recognition.** Anki's `{{c1::answer}}` cloze requires the user to recall and type the answer before revealing, which is stronger than 4-choice recognition. Anki data shows typing-recall produces ~30% better long-term retention vs. MC recognition for the same card.

## B. Pickup 現狀 (對齊 Anki lens — 5 bullets)

- **SRS queue is a ghost.** `readSrsQueue()` in `src/data/storyKitten.ts:376` maintains an array of wrong-answer question IDs, but zero UI surface in `TasksPage`, `MapPage`, or `ProfilePage` ever reads or displays it. A user could have 40 struggling cards and see no signal anywhere.
- **SRS activates silently inside chapter run.** `runStore.ts:328` injects up to 3 SRS cards at the start of a story chapter — but they appear identically to new questions. No "複習" label, no different treatment. Users never know they are drilling prior mistakes.
- **Binary remove-on-first-correct.** `removeFromSrs` fires on the *first* correct answer (`runStore.ts:429`). A card answered wrong 5× and then correct once is graduated — same as a card wrong once. No interval, no ease factor, no "correct twice = mature" gate. Retention is structurally overestimated.
- **No dedicated review-mode entry point.** Anki's core UX loop is: "open app → see due count → tap deck → rate cards → done." Pickup has no equivalent. There is no button to say "drill my struggling cards now." The SRS capability exists in code but is reachable only as a silent side-effect of starting a story chapter.
- **No leech detection or hard-card labeling.** Nothing in `addToSrs` or `srsReviewBatch` tracks how many times a given ID has been added. A card that has been wrong 12 times looks the same as one wrong once. No escalation path, no suspension, no "you've missed this 3 times — here's why" moment.

### 2. CompletePanel SRS-growth notification
**Where:** `src/react-app/pages/LessonPage.tsx` → `CompletePanel`
**What:** `answerLog.current` already contains all `isCorrect` flags. Count wrongs, display if > 0:
```tsx
const newSrsCards = log.filter(a => !a.isCorrect).length;
// Under the 3-stat row:
{newSrsCards > 0 && (
  <div style={{ fontSize: 13, color: '#c84a3a', fontWeight: 700, marginBottom: 12 }}>
    🗂️ {newSrsCards} 個難題加入複習庫 · added to your review queue
  </div>
```

Closes the feedback loop Anki makes explicit: "you struggled with 3 cards, they'll come back."

### 3. Require 2-correct to graduate SRS (upgrade from 1×)
**Where:** `src/data/storyKitten.ts` → `removeFromSrs` + `runStore.ts` → `answer()`
**What:** Add per-question correct-streak counter in localStorage:
```ts
// storyKitten.ts additions
const LS_SRS_CC = (id: string) => `pickup.srs.cc.${id}`;
export function incrementSrsCorrect(id: string): number {
  const n = Number(localStorage.getItem(LS_SRS_CC(id)) ?? 0) + 1;
  localStorage.setItem(LS_SRS_CC(id), String(n));
  return n;
}
export function resetSrsCorrect(id: string): void {
  localStorage.removeItem(LS_SRS_CC(id));
}
```
In `runStore.ts answer()`: call `incrementSrsCorrect(round.id)` on correct; only call `removeFromSrs` when return value ≥ 2. Reset on wrong with `resetSrsCorrect`. Structurally prevents the "lucky guess graduates the card" false-positive that Anki's SM-2 avoids through the ease factor mechanism.
