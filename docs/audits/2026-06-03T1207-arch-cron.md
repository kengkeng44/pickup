## D. i18n Architecture Risk (angle-specific, 5 bullets)

### The current i18n strategy

Pickup uses **"bilingual by convention"** — no i18n library, no locale files. Strings are written inline as `zh · EN` (e.g., `'答對 · Nice!'`). Content schema has explicit bilingual fields: `titleZh`/`titleEn`, `explanationZh`, `optionsZh`. `index.html` sets `lang="zh-TW"`.

This is intentional and works for the current single-market release. Five architectural risks:

1. **Chapter title data lives in 4 independent locations** — `src/data/storyKitten.ts`, `src/react-app/pages/MapPage.tsx`, `src/react-app/pages/ChaptersPage.tsx`, `src/react-app/views/MapView.tsx` all declare the same `{ titleZh, titleEn }` objects for all 8 chapters. 10 duplicated declarations found. Any chapter title change requires editing 4 files. Ch2-8 expansion will compound this — if `titleZh` for Ch3 changes, 3 out of 4 sources will silently drift. No test guards this.

2. **64 UI string literals scattered across 14 React files** — No central string table. Key examples:
   - `載入中…` (loading state) appears in `LessonPage.tsx` and `ChaptersPage.tsx`-equivalent rendered divs
   - Nav labels `['首頁','任務','我的','成就']` live in `BottomNav.tsx` only — OK, but not type-checked or testable
   - `今日目標`, `每日任務`, `連續學習至少 1 天` in `TasksPage.tsx` are not linked to any content spec
   - `renderers.tsx` has 42 lines of non-ASCII: `🎧 點喇叭聽完聲音再選答案`, `聽聲音, 打出你聽到的句子`, `選字排這裡…`, `清除`, `送出`, `全部配對完成` — all render-critical, all untestable as strings

3. **`lang="zh-TW"` mismatch** — `index.html` declares the document language as Traditional Chinese, but 60%+ of rendered content is English (lesson sentences, question text, options). Screen readers (VoiceOver on iOS) will attempt to read English lesson sentences with a Chinese voice engine, causing severe mispronunciation. The WCAG 3.1.1 Language of Page criterion requires `lang` to match the *primary* content language. An English-learning app's primary content is English; the UI chrome is zh-TW. Correct pattern: `<html lang="en">` with Chinese UI elements marked `lang="zh-TW"`.

4. **KR/JP expansion requires touching 950 lines across 58 files** — `docs/product/pickup-pm-roadmap-2026-06.md` plans Q2-2 2027 Korean/Japanese display language. The roadmap acknowledges "~5000 strings rewrite." With the current inline pattern, a translator cannot work on a locale file — they must edit TypeScript source. No extraction tooling (i18next-parser, formatjs) exists. The actual rewrite surface is 950 source lines with Chinese, spanning 58 files across 4 layers (React pages, legacy UI, data, scenes). Without a string extraction pass first, KR/JP is blocked at the architecture level.

5. **PRAISE/feedback microcopy pool is the one well-structured exception** — `src/ui/ClozeUI.ts` has a single `PRAISE_CORRECT` array (8 bilingual strings) and `PRAISE_WRONG` array (6 strings). This is the right pattern. The risk: `renderers.tsx` defines its own one-off praise-style strings (`'She shakes her head ✨'`, `'Not that one ✨'`) inline in a component rather than importing from the pool. Two parallel praise systems will drift in tone/style as content grows.

## E. Top 3 P0 Architecture Risks

**P0-1: Chapter title defined in 4 places — schema deduplication needed**

`src/data/storyKitten.ts` + `MapPage.tsx` + `ChaptersPage.tsx` + `MapView.tsx` each declare the same `{ titleZh, titleEn }` objects. Current count: 10 duplicated `titleZh` declarations across 4 files. Fix: extract a single `src/data/chapters.ts` constant exporting `CHAPTER_META: Record<number, { titleZh, titleEn, accent }>`. All 4 consumers import from it. Zero risk, 2-hour task, eliminates future silent drift as Ch2-8 content lands.

**P0-2: `lang="zh-TW"` accessibility violation on English-content app**

VoiceOver (iOS, primary Pickup target device) reads `lang="zh-TW"` document and applies Mandarin TTS to all text including English lesson sentences. Users relying on screen readers hear `"I jump on the wall again this evening"` read in Mandarin phonemes. Fix: flip `index.html` to `<html lang="en">`, then apply `lang="zh-TW"` to individual zh-UI elements (`<span lang="zh-TW">首頁</span>`, etc.) or wrap nav/UI chrome in a `<div lang="zh-TW">`. This is a WCAG 3.1.1 compliance issue that would block App Store review under Apple's accessibility guidelines.

**P0-3: 11-test regression since last cron — scope unknown**

Test count dropped 34 → 23 between 2026-06-01T1633 and now. No commit message in the cron audit log explains this. Missing tests could mean: deleted test files, test files excluded from `vitest.config.ts`, or failing tests that are not reporting. `npm run test -- --reporter=verbose` should be run to confirm the missing 11 are intentionally removed, not silent failures.
