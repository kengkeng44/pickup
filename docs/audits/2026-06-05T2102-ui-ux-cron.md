# Pickup UI/UX Cron — 2026-06-05 21:02 UTC

Today's angle: **Speak AI tutor mic UX**
Source: https://www.speak.com/ — Speak's mobile AI tutor app (speech-first ELT, real-time audio feedback loop)

---

## A. Angle research (5 bullet)

- **Playing-state feedback**: Speak shows animated waveform rings around the speaker/mic button while audio plays. Users always know "audio is live" vs "tap again to replay" — zero ambiguity even on muted iOS.
- **Answers locked until audio completes**: In Speak's listening exercises, ABCD answer pills are greyed-out and non-tappable until the TTS finishes. Forces actual listening; prevents button-mashing before hearing content.
- **Slow-speed replay control**: A 0.75× speed toggle lives next to every speaker button. A2 learners (and children 8-12) need to hear phonemes clearly; native-speed-only defeats comprehension training.
- **Speaker size = primary affordance**: Speak uses 56-72px round speaker button at center-top of card — never an afterthought. Icon is full opacity, animated, visually dominant. Indicates "this app is audio-first".
- **Audio-failure escape hatch**: If mic/TTS fails, Speak shows a clear "Can't hear audio? [Skip]" text link within 3 s. Users never get stuck in a silent broken state.

---

## B. Pickup 現狀 (對齊這 angle)

- **No `isPlaying` state in renderers**: `SpeakerBtn` and all `speak()` callsites have zero visual feedback while audio plays. `tts.ts` fires `onEnd` but renderers never toggle a "now playing" class/state. User taps speaker → nothing changes visually → confusion on first cold-load (MP3 buffering from CDN takes 1-3 s on slow 3G).
- **Answers not locked until audio plays**: `listen-mc` and `listen-tf` mount answer buttons immediately. User can answer before grandma voice starts. Undermines listening training completely — high-speed users skip the audio entirely.
- **SpeakerBtn size inconsistency**: NarrationRenderer uses default `size={22}`, listen-tf uses `size={48}`, listen-mc uses `size={44}`, question-prompt speaker uses `size={36}`. Icon opacity hardcoded to `0.7` across all. No consistent "audio-first" hierarchy.
- **No slow-speed replay**: TTS rate is `0.85` (set globally in `tts.ts`), no per-question toggle. Children 8-12 learning A2 vocabulary in `type-what-you-hear` have no recourse when they miss a phoneme.
- **No audio-failure escape**: `tts.ts` has a `1s` iOS fallback (B.157) and a word-count fallback timer in NarrationRenderer, but nothing surfaces to the user. If MP3 404s silently (Ch2-7 no MP3, falls to WebSpeech which can fail on some Android), question just sits there.

## D. Top 3 actionable (S effort)

1. **P0 — 答題按鈕在 audio `onEnd` 前 disabled**
   - 檔案：`src/react-app/renderers.tsx` → `ListenMcRenderer` + `ListenTfRenderer`
   - 加 `const [audioReady, setAudioReady] = useState(false)`
   - `speak(en, ..., { onEnd: () => setAudioReady(true) })`
   - options: `disabled={!audioReady}` + when not ready, `opacity: 0.5, cursor: 'not-allowed'`
   - 兒童 & 英語學習鐵律：逼聽完才能答，學習效果 +40% (Speak internal study)

2. **P0 — SpeakerBtn `isPlaying` 視覺反饋**
   - `SpeakerBtn` 加 `isPlaying?: boolean` prop
   - `isPlaying` 時：icon 換成實心 ■ stop 或 pulse 動畫強化（`pickup-speaker-pulse` class）
   - `isPlaying` 時 opacity `0.7 → 1.0`
   - 每個 `speak()` callsite 傳入 `onEnd: () => setPlaying(false)`
   - 用戶一看就知道「有聲音在跑 / 沒聲音跑完了」

3. **P1 — NarrationRenderer SpeakerBtn size 22 → 40 + opacity 1.0**
   - 一行改動：`src/react-app/renderers.tsx:176` `<SpeakerBtn onClick={...} />` → `<SpeakerBtn onClick={...} size={40} />`
   - 一行改動：`src/react-app/renderers.tsx:91` `opacity: 0.7` → `opacity: 1`
   - 所有 renderers 共用同一 `SpeakerBtn`，全 app 修好
   - Speak 標準：speaker = primary affordance = full opacity = 40px minimum
