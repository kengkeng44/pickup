## A. Angle research (5 bullet)

- **Playing-state signal**: Speak's speaker/mic button flips to a live waveform SVG while audio streams. Users always know: is audio playing, buffering, or finished? The icon is never ambiguous. Absence of state = silent failure = user re-taps = double play.
- **Replay affordance with count**: Speak and Pimsleur both surface "listened N time(s)" micro-label next to the replay button; seeing "聽了 2 次" primes users to feel confident before answering. Research (Tandfonline 2025, playback controls in TELL) shows beginner L2 learners replay audio 2-3× before answering dictation-cloze — a replay-count affordance directly matches this behavior.
- **Load-state vs playing-state distinction**: Speak shows a loading ring (spinner inside button) while fetching TTS; transitions to waveform once playback begins; returns to static icon on completion. Three distinct states. Language learners misattribute silence to "audio unavailable" when only state is static icon.
- **Speaker icon prominence**: Speak and Duolingo both use full-opacity, large (48-56px) icons with a cream/light fill background — not transparent or muted. Opacity < 1 is a standard "disabled" signal in mobile UI; the Speak icon never goes below opacity 0.9.
- **Audio chaining transparency**: In Speak role-play, text and audio arrive in sequence with a brief ellipsis "..." bubble showing "thinking" between AI turns. This prevents the user from tapping during the gap and interrupting queued audio. Pimsleur uses a visible delay bar ("▸ coming up…") between prompts.

## B. Pickup 現狀 (對齊這 angle)

- **No playing-state indicator**: `SpeakerBtn` renders a static 22-44px `icon-speaker.webp` at `opacity: 0.7` regardless of whether audio is actively playing, loading a buffer (up to 800ms first-fetch), or finished. User has zero visual confirmation audio started. iOS silent failure (unlock not yet triggered) is invisible.
- **opacity: 0.7 = looks disabled**: Every `SpeakerBtn` instance and the `NarrativeLine` replay button uses `opacity: 0.7`. This is the standard "disabled" signal in Material + iOS HIG. It tells users "this button may not work" before they tap.
- **Audio chaining gap is invisible**: `listen-mc` / `listen-tf` chains `sentence MP3 onEnd → 400ms setTimeout → speak(qPrompt)`. During that 400ms gap — and during the async `loadBuffer()` network fetch before first play — the UI is silent and static. User cannot tell if audio is coming.
- **type-what-you-hear: no replay count cue**: The instruction is static: "聽聲音, 打出你聽到的句子". No indication of how many times the user has heard it, no encouragement to replay before typing. Beginner learners (A2 target) need 2-3 listens; the current UI does not nudge this.
- **listen-tf SpeakerBtn size inconsistency**: Pre-reveal = size 48, post-reveal = size 36. Same content, different affordance size = visual inconsistency. User expects tap target to be stable across the interaction.
- **No first-load skeleton**: When `preloadLessonAudio()` hasn't resolved (common on first lesson start, Ch2-8 with no MP3 cache), `speak()` has a 500-800ms async `loadBuffer` delay after tap. SpeakerBtn shows nothing. This is indistinguishable from failure.

## D. Top 3 actionable (S effort)

**D1 — SpeakerBtn opacity fix (P1, XS)**
`src/react-app/renderers.tsx` 的 `SpeakerBtn` 組件: `opacity: 0.7` → `opacity: 1.0`。同步修 `src/react-app/pages/LessonPage.tsx` `NarrativeLine` replay button 的 `opacity: 0.7`。零設計討論，零架構影響。

**D2 — type-what-you-hear replay count cue (P1, S)**
`TypeWhatYouHearRenderer` 加 `replayCount` state (初始 0)，每次手動點 SpeakerBtn 時 `+1`，`useEffect` 自動播不計入；SpeakerBtn 下方加 `replayCount > 0 && <span style={{fontSize:11, color:'#b07a2a'}}>已聽 {replayCount} 次</span>`。符合 Speak + Pimsleur「heard N times」pattern，直接對齊 A2 學習者 2-3× replay 行為。

**D3 — listen-tf SpeakerBtn size 一致化 (P2, XS)**
`ListenTfRenderer` pre-reveal `<SpeakerBtn ... size={48} />` → `size={44}`; post-reveal `size={36}` → `size={44}`. 一致的 tap-target 減少認知跳躍。再加 pre-reveal 的 container `border: '2px solid #e7a44a'` 圓形背景, 讓大 icon 有視覺錨點（對齊 Speak 的 framed mic button pattern）。
