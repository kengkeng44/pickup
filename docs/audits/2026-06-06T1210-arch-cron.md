## E. Top 3 P0 architecture risks

### P0-1 — `streak.updateStreak()` zero-tested, silent D7 bleed
`updateStreak()` is called on every app boot (`BootScene.ts`) and is the primary D7/D30 retention signal. The `gap ≥ 2 → reset to 1` branch runs when users miss a day — the most common path post-Day-7. No test exists. A date string comparison bug (e.g. `daysBetween` using `T00:00:00` suffix for local time but `isoDate()` producing local date while test environment is UTC) would cause false resets in CI passing silently. Impact: streaks reset → users feel punished → churn.

**Cheapest fix:** 10-line test with `vi.stubGlobal('localStorage', ...)` (same pattern as lesson-progress.test.ts) + `vi.setSystemTime` to simulate day gaps.

### P0-2 — ProfilePage LS key split: silent name customisation loss
`ProfilePage.tsx:18` writes `localStorage.setItem('pickup.catName', v)`.
`catName.ts:9` reads from `'pickup.cat-name'` (hyphen, not camelCase).
These are different keys. Any user who opens Profile → sets their cat's name → goes to lesson will see `Mochi` everywhere regardless of input. The Profile input appears functional but writes to a dead key.

Additionally, R10 spec says "Profile customization removed" (B.148) yet the UI still renders. This is a two-layer bug: spec drift (UI not removed) + code bug (wrong LS key).

**Cheapest fix:** Change ProfilePage line 13+18 to use `'pickup.cat-name'`. Or remove the input if R10 "retired" is still the intent (spec is ambiguous — LESSON_HOOKS still have `{catName}` in storyKitten.ts load path).

### P0-3 — `narration` (53% of Ch1) schema definition untested
The `NarrationSchema` definition is `{ ...QuestionBaseFields, type: 'narration' }` — the simplest schema in the union. But `QuestionBaseFields` contains `level`, `difficulty`, `explanationZh`, and optional `tags`. A PR that adds a required field to `QuestionBaseFields` (e.g. `speaker: z.string()`) would make all 41 Ch1 narration entries fail Zod parse at runtime, showing a blank lesson. The `lessons-ch1-validate.test.ts` parses the JSON — but that tests the JSON data file, not the schema definition. If the schema definition changes to require a field the JSON already provides, CI passes; the real danger is a schema *addition* that breaks existing JSON without updating the JSON.

**Cheapest fix:** Add `it('accepts valid narration entry', ...)` to lessons.test.ts — 6 lines.
