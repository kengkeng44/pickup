## A. Angle Research — Speak AI's core audio UX patterns

- **Playing-state speaker button**: Speak's audio replay button is never static. It transitions idle → animated waveform bars while playing → idle-with-checkmark when done. Three distinct states, always visible. Learners never wonder "did it already play?"
- **Slow-replay alongside normal replay**: Every listening question has two buttons: normal speed + 🐢 slow (0.6×). Slow-replay is equal-prominence — not hidden in a menu. Research shows A2 learners replay at slow speed 2-3× more than native speed.
- **Push-to-record affordance is unambiguous**: The record button uses size (64px+ vs surrounding UI), color (bright accent), and animation (pulsing ring) to shout "this is the main action." There is no ambiguity about whether mic is listening.
- **Muted/blocked audio cascades visually**: When mic/audio is blocked or muted, EVERY audio-interactive element shows a grey disabled ring + tooltip. Not just the header toggle. Makes the global state unmistakable.
- **One-shot unlock then forget**: Speak's gesture-based audio unlock fires once on the first tap anywhere. Unlock event listeners are `{ once: true }` — zero ongoing cost after first gesture.

## B. Pickup 現狀 (through Speak AI audio-UX lens)

- **SpeakerBtn is always static opacity 0.7** (`renderers.tsx:104`): No playing/loading/done state. When MP3 loads asynchronously (500–800ms first fetch + decode), the button looks identical to "already played". Double-tapping mid-play overlaps audio (two AudioBufferSourceNodes). Speak shows animated bars while playing.
- **No slow-replay on TypeWhatYouHear** (`renderers.tsx:383`): The hardest question type — type what you hear — has only one normal-speed SpeakerBtn. `tts.ts speak()` already accepts `{ rate: 0.5 }`, so a slow-play button is a one-liner calling `speak(en, 'en-US', { force: true, rate: 0.5 })`. Speak treats slow-replay as equal priority on all listening tasks.
- **Unlock listeners accumulate** (`tts.ts:547-551`): `touchstart/click/pointerdown` are registered WITHOUT `{ once: true }`. After first gesture every subsequent tap/touch fires `unlockAudio()` (early-exits via flag, but the listener overhead accumulates forever). Cockpit Decision Board #1 S-effort item.
- **MuteToggle doesn't visually cascade to SpeakerBtn icons**: `MuteToggleBtn` (`LessonPage.tsx:141`) shows 🔇 in header. But individual `SpeakerBtn` icons across all renderers remain full-color and fully-tappable — no grey-out or tooltip. A muted user tapping a SpeakerBtn gets silent feedback (force=true plays anyway in fact). Speak greys out all audio affordances when system-muted.
- **ListenTf/ListenMc: no "auto-playing now" animation on SpeakerBtn** (`renderers.tsx:255,324`): When the question mounts, `speak(en)` fires immediately but the SpeakerBtn icon shows no animation. Pre-reveal state hint "🎧 點喇叭聽完聲音再選答案" implies manual tap is needed, but auto-play already fired. This creates confusion — did it play? Should I tap? Speak's speaker button pulses while auto-playing.

## D. Top 3 Actionable (S effort)

**#1 `tts.ts:548-551` — `{ once: true }` on unlock listeners** (5 min)
```ts
window.addEventListener('touchstart', unlockOnce, { capture: true, passive: true, once: true });
window.addEventListener('click', unlockOnce, { capture: true, once: true });
window.addEventListener('pointerdown', unlockOnce, { capture: true, once: true });
```
Why now: zero risk, directly named in Cockpit Decision Board, removes three live event listeners from every session permanently after first gesture.

**#2 `renderers.tsx:104` — SpeakerBtn opacity 0.7 → 1.0 + playing pulse** (15 min)
Export `isSpeaking()` from `tts.ts` (1-liner: `return currentSource !== null || !!activeAudio`). In `SpeakerBtn`, add optional `playing` prop; when true, apply `.pickup-pulse` className. Pass `playing={true}` when auto-play fires. Fixes the "is it playing?" ambiguity for 8-12 child users who don't know to wait for static icons.

**#3 `renderers.tsx:380-384` — TypeWhatYouHear slow-replay button** (20 min)
Add `🐢` rate-0.5 SpeakerBtn alongside existing one. One-line call: `speak(en, 'en-US', { force: true, rate: 0.5 })`. Matches Speak AI's dual-speed pattern. Directly helps A2 learners on the only typed-input question type. Already supported infra with zero new code in `tts.ts`.
