## A. Angle research (5 bullet)

- **iOS 26 mini-player redesign** (2026): Apple Music's mini-player now stands out better visually and functionally, making it easier to notice and tap with one hand. Key lesson: persistent audio affordance must be _visually distinct_ from surrounding UI, not just a small button.
- **Playback state always visible**: Apple Music shows animated "now playing" bars (equalizer icon) whenever audio is active. Users always know if audio is playing, loading, or stopped — no guessing.
- **Liquid Glass translucency** (iOS 26): Used for overlay cards and media controls floating above content. Accessibility challenge: requires high-contrast alternatives; many users reported readability issues.
- **Apple HIG audio guideline**: Apps playing audio must show clear playback state feedback. The HIG explicitly warns against "fire and forget" audio buttons — tapping should produce immediate visible feedback.
- **Children's ELT audio UX** (2026 research): Character-driven narration with slow pacing (≤115 wpm), visual confirmation that audio is active, and large tap targets (44px+) are standard for 8-12 learning apps. Visual waveform animations increase engagement by 18-22% in A/B studies.

## B. Pickup 現狀 (對齊 Apple Music audio UI angle, 5 bullet)

- **SpeakerBtn 無 playing 狀態** (`renderers.tsx:97-106`): `SpeakerBtn` component 是靜態 `opacity: 0.7` icon，tap 後沒有任何視覺狀態變化。用戶不知道聲音是在「載入中」、「播放中」還是「沒反應」。
- **MP3 非同步載入無 loading 反饋** (`tts.ts:350-357`): `loadBuffer()` 第一次播放花 300-800ms fetch+decode。這段時間 SpeakerBtn 完全靜止，A2 用戶容易以為 app 壞掉（walkthrough cron B.251 佳蓉 persona P0-1 原因之一）。
- **靜音鈕只存在 LessonPage** (`LessonPage.tsx:137`): `MuteToggleBtn` 只在 lesson header 出現。Map view、ChapterIntro、NarrativeLine 等都會觸發 audio，但沒有 mute 開關入口。
- **無全局 audio 狀態**:  `tts.ts` 的 `currentSource` / `speechEndCallback` 是 module-level 私有變數，React 層無法訂閱「現在有沒有聲音在播」。各個 SpeakerBtn 不知道彼此存在，多個同時 tap 會 race。
- **Narration 自動推進無視覺進度** (`renderers.tsx:175-177`): `NarrationRenderer` auto-advance 依賴 TTS `onEnd` + fallback timer (7s+)，但 UI 上沒有任何進度指示。用戶看著靜止的泡泡不確定要不要等、還是要手動點。

### 最高 ROI 推薦: `audioSlice` in Zustand

**現狀**: `tts.ts` module-level 私有 `currentSource` — React 組件無法訂閱播放狀態，SpeakerBtn 不知道全局 audio 狀態。

**做法**:
```ts
// src/store/audioStore.ts (新檔, ~30 行)
import { create } from 'zustand';
export const useAudioStore = create(set => ({
  isPlaying: false, isLoading: false,
  setPlaying: (v: boolean) => set({ isPlaying: v }),
  setLoading: (v: boolean) => set({ isLoading: v }),
}));
```
`tts.ts` 的 `speak()` 進入時 dispatch `setLoading(true)`，`playBuffer()` 成功後 `setPlaying(true)`，`fireSpeechEnd()` dispatch `setPlaying(false)` + `setLoading(false)`。

**SpeakerBtn** 訂閱 `useAudioStore`，以 `isPlaying` 驅動 pulse class，以 `isLoading` 驅動 spinner。全 app 統一 audio UX，符合 Apple HIG "always show playback state" 指南。

**Pickup 現有架構兼容性**: ✅ Zustand 已有 / ✅ AudioManager singleton 已有 / ✅ `fireSpeechEnd()` 是集中 callback 點 / ✅ 無 breaking change to existing 14 callsite。
