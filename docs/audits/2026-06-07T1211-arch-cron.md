## D. Angle-specific risk — SOLID compliance (5 bullets)

**D1 — SRP: `LessonScene.ts` is a god class (1,475 lines, 5 concerns)**

Single file simultaneously handles:

| Concern | Evidence |
|---------|---------|
| Question rendering (3 inline custom renderers) | `_renderNarration`, `_renderListenTf`, `_renderListenTfZh` — 280+ lines of inline DOM string templates |
| Answer logging + accuracy stats | `lessonAnswerLog`, `_snapshotTf/TfZh/AnsweredQ` — 32 usages |
| TTS/audio orchestration | 79 `speak`/`audio`/`bgm` references; duck/unduck calls woven through render paths |
| Completion overlays (stats + review) | `_showLessonStats` 170 lines, `_showLessonReview` 128 lines — both fully owned by this class |
| Timer / advance scheduling | `scheduleAdvance`, `cancelAdvanceTimer`, `Phaser.Time` — 16 usages |

A change to the answer accuracy display requires opening the same file as a change to narration auto-advance timing — no separation of concerns.

**D2 — OCP: `renderQuestion()` is closed to extension (if/else type dispatch)**

`LessonScene.renderQuestion()` at line 684 dispatches question types via nested `if/else` chains:

```
if qType === 'narration' → _renderNarration()
else if qType === 'listen-tf' → _renderListenTf()
else if qType === 'listen-mc' || 'listen-comprehension' || ... → ClozeUI path
else if qType === 'tap-tiles' || 'tap-pairs' || ... → TapInputUI path
```

Every new question type requires *modifying* this function. The React path (`LessonPage.tsx:90`) uses an open-to-extension map:

```ts
const Renderer = RENDERERS[q.type] ?? FallbackRenderer;
```

`RENDERERS` covers 15 types; `LessonScene` inline-handles 7. The Phaser path will never catch up without a renderer-map refactor — each new type widens the divergence.

**D3 — SRP: `tts.ts` owns three distinct audio subsystems in one 544-line file**

| Subsystem | Lines |
|-----------|-------|
| MP3 pre-recorded path (indexLookup, playBuffer, preload*) | ~230 lines |
| WebSpeech fallback (speakWebSpeech, fallbackWebSpeech, fireSpeechEnd) | ~80 lines |
| iOS unlock chain (unlockAudio, warmUp*, IS_IOS_DEVICE) | ~80 lines |

All three export through a single `speak()` entry point. The mixing means: adding a new audio backend (e.g., Capacitor native audio in Phase 2.5) requires editing the same file that contains iOS quirk workarounds and WebSpeech rate tuning. The subsystem boundaries are invisible to callers.

**D4 — DRY: `wrapWordsForHint()` / `wrapWords()` identical implementations in two files**

`LessonScene.ts:17-27` and `renderers.tsx:53-64` are character-for-character the same function (local variable name differs: `escape` vs `esc`). The duplication is structural — Phaser scene can't import from React renderer — but it means a bug fix (e.g., unicode punctuation not wrapped) must be applied in two places independently. This already happened once: the `&#039;` apostrophe escape was added to both, but future changes risk diverging silently.

**D5 — Dual-path architectural drift: Phaser `LessonScene` vs React `LessonPage` implementing parallel lesson engines**

The codebase maintains two complete lesson-execution engines:

| Feature | LessonScene (Phaser) | LessonPage (React) |
|---------|---------------------|-------------------|
| Question dispatch | 7 types, if/else chain | 15 types, RENDERERS map |
| Answer log schema | `{q: any; userIdx; correctIdx; isCorrect}` | `{q: RawQuestion; userIdx; isCorrect}` |
| Completion UI | `_showLessonStats()` (170-line inline DOM) | `<CompletePanel />` (React component) |
| TTS calls | Direct `speak()` in render methods | Per-renderer `speak()` in useEffect |

Both paths are live in production (`bootGame.ts` loads Phaser + `LessonScene`; `App.tsx` routes to `LessonPage`). Schema differences between `lessonAnswerLog` (has `correctIdx`) vs React `answerLog` (lacks `correctIdx`) mean the two engines produce incompatible analytics payloads. Phaser path also has `_showCompletionArticle` dead code (R5) that React path never implemented.

## E. Top 3 P0 architecture risks

### P0-1 — Dual-path drift will produce silent data inconsistency (Phaser vs React lesson engines)

Both `LessonScene` (Phaser) and `LessonPage` (React) call `track(EVENT.LESSON_ANSWER, ...)` with different payload shapes. `LessonScene` includes `correct_idx: correctIdx` (line 521); `LessonPage` does not (line 105 only sends `user_answer_idx` and `is_correct`). PostHog events from the two paths are not homogeneous. Funnel queries on `correct_idx` will silently undercount all React-path sessions.

Additionally, `runStore.markLessonCompleted()` is called from both paths — but `LessonPage` gates it on `isLastLessonOfChapter` (line 82) while `LessonScene.finish()` calls it unconditionally. Players who reach the last lesson on each path will have different unlock states if the flag logic differs.

**Cheapest fix:** Decide which path is canonical for Ch1 (React is the v2.0 direction per `App.tsx` routing). Audit `bootGame.ts` to confirm Phaser `LessonScene` is only reachable via legacy routes. Add E2E or integration smoke test on lesson-complete event shape.

### P0-2 — R5 spec vs implementation mismatch: `_showCompletionArticle()` is dead but spec mandates it

The canonical spec v2 R5 states: *"When advance() reaches end of questions[], `_showCompletionArticle` fires — concatenates all narration entries' sentence into clean paragraph + THE STORY header + replay + Done CTA."*

Actual: `finish()` at `LessonScene.ts:1048` calls `_showLessonStats()`. `_showCompletionArticle()` is a 86-line dead method explicitly suppressed with a `void` cast. The React path (`CompletePanel`) shows XP/Accuracy/Time — no story paragraph.

This means: no player currently sees the story-paragraph completion article. The feature is spec'd, partially implemented (dead code exists), and absent from product. Either:
- Spec R5 needs to be updated to document `_showLessonStats` as the canonical completion (current reality), OR
- The story-paragraph feature needs to be wired in (completes the Duolingo Stories reading loop)

Either way, an uncommitted orphan spec creates confusion for the next agent/dev opening this file.

### P0-3 — `LessonScene` god class is one question type away from being unmaintainable

`LessonScene.ts` is already 1,475 lines with 5 concerns, 7 inline type handlers, and 79 audio callsites woven through every render path. The React path's `RENDERERS` map (15 types, extensible) shows the correct pattern. When Ch2-Ch8 content ships with new question types (`listen-build`, `drag-blank`, `speak-back`, `picture-mc`) — all present in `RENDERERS` but absent from `LessonScene` — either:

1. New types silently fall through to the ClozeUI fallback (incorrect behavior), or
2. Developers extend `LessonScene.renderQuestion()` with more if/else branches — accelerating the god-class problem

Since `LessonScene` is the Phaser-path lesson engine still loaded by `bootGame.ts`, a missing type handler is a silent content regression. The React path already handles all 15 types correctly — the safer long-term fix is confirming `LessonScene` is only a legacy shim and preventing new lesson content from routing through it.
