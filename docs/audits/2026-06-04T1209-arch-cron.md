## D. Performance budget drift — angle-specific risk (5 bullets)

### 1. 39 MB of source PNGs deployed to production with 1-day CDN cache

`public/mascots/preview/` contains 28 raw source PNG files (1.3–1.7 MB each, total 39 MB). Vite copies all `public/` contents verbatim to `dist/`. These files are:
- Served at `https://pickupwords.pages.dev/mascots/preview/calico-anchor-v1.png` etc.
- Covered by `_headers` rule `/mascots/*: Cache-Control: public, max-age=86400, stale-while-revalidate=604800` — any crawl or accidental URL visit caches a 1.5 MB PNG in the browser for 1 day
- Covered by the service worker `/mascots/*` pattern, but the SW's `size > 500 * 1024` guard correctly rejects caching them (however it still fetches them on first request)
- Cloudflare Pages deploy quota is consumed by these files on every `wrangler pages deploy` — at 39 MB per deploy, a 10-deploy sprint wastes 390 MB of bandwidth

**Fix**: Move `public/mascots/preview/` to `tools/raw-mascots/preview/` (consistent with v1.9.30 audit-9 precedent). Zero source-code changes needed.

### 2. Incorrect `fetchpriority="high"` on LCP-wrong image

`index.html` preloads 6 images with `coin-gold.webp` (178 KB) tagged `fetchpriority="high"`. On MapPage, `coin-gold.webp` renders as a 26×26 px HUD icon — it is **not** the LCP element. The visible-above-fold LCP candidates are `iso-grandma.webp` (126 KB, ~88 px wide character) and the map banner. `coin-gold.webp` steals the high-priority bandwidth slot, delaying the actual LCP image by one parallel download slot on slow mobile connections.

**Total preloaded image weight**: coin-gold (178 KB) + crown-gold (115 KB) + iso-grandma (126 KB) + iso-shiba (96 KB) + icon-paw (123 KB) + icon-flame (73 KB) = **711 KB**. On 3G (128 KB/s ≈ 1 Mbps) this takes ~5.5 s before the LCP image finishes loading — violating R12 (≤ 3000 ms) by >80%.

**Fix**: Swap `fetchpriority`: set `iso-grandma.webp` to `fetchpriority="high"`, demote `coin-gold.webp` to no-priority (or remove its preload entirely since it only appears in HUD after React mounts).

### 3. `modulePreload: false` eliminates chunk prefetch, forcing waterfall loads

`vite.config.ts` sets `build.modulePreload: false`. This disables Vite's automatic `<link rel="modulepreload">` hints for split chunks (zod, react-router, zustand). On first navigation to a lesson:
1. Main index.js parses and executes
2. `React.lazy(() => import('./pages/LessonPage'))` fires — browser only then discovers it needs LessonPage.js
3. LessonPage.js fetches and parses — then triggers further imports
4. No parallel hint was available to start these downloads earlier

On a 3G connection, each waterfall step adds ~300 ms RTT. The 3-step chain (index → LessonPage → sub-imports) adds ~900 ms before the lesson renders. Without `modulePreload`, the full lesson LCP on first visit is: HTML (1 RTT) + JS chunks waterfall (2–3 RTTs) + React mount + image fetch = likely 3–5 s on 3G.

The comment in vite.config says B.151 disabled modulePreload but gives no reason beyond "see git history." This should be re-evaluated: `modulePreload: { polyfill: false }` (no polyfill, keep the hints) is the correct setting for modern browsers.

### 4. POV scene images served as PNG with no cache header and no WebP

`public/pov/ch1-q*.png` range 83–351 KB each and are referenced as CSS `background-image` in `src/style.css`. Two problems:
- **No cache header in `_headers`**: `_headers` has rules for `/mascots/*`, `/audio/*`, `/assets/*` but nothing for `/pov/*`. Cloudflare will apply its default heuristic caching (~4 hours) — far shorter than the 1-day explicit rule for mascots. Users returning to Ch1 lessons re-download 1.1 MB of POV images.
- **PNG not WebP**: `ch1-q2.png` is 351 KB. Equivalent WebP would be ~100–150 KB (55–70% reduction). Ch2–8 will add more scene images; each POV PNG will inflate the repeat-visit payload.

**Fix 1**: Add `/pov/* Cache-Control: public, max-age=86400, stale-while-revalidate=604800` to `_headers`.
**Fix 2**: Convert PNGs to WebP (`cwebp -q 85`) — reduces 1.1 MB → ~350–400 KB.

### 5. `public/` contamination with dev tooling causes 11.5 MB of CDN bloat per deploy

12 HTML files + 1 Markdown file live in `public/`:
- `cockpit.html` (671 lines / 25 KB), `dashboard.html` (801 lines / 33 KB), `qa-dashboard.html` (648 lines / 31 KB), `qa-static-ch0.html` through `qa-static-ch7.html` (270–371 lines each), `preview-ch1-first25.html` (314 lines / 17 KB), `qa-static.html`
- `todo-永久token設定.md` (9.5 KB) — describes Infisical token architecture for wordwar + cheng.robot repos; live at a guessable URL

These files have no `Cache-Control` rule (falls through to browser default). They consume ~2 MB per deploy cycle and expose internal tooling structure. `todo-永久token設定.md` is a moderate information-disclosure risk: it does not contain live credentials but maps the full secret management topology (Infisical → Cloudflare API → Railway secrets).

**Fix**: Move all dev HTML files to `tools/dev-dashboards/` (not under `public/`). Move `todo-*.md` to `docs/`. Add `.gitignore` or Vite config to exclude them from `public/` entirely.

## E. Top 3 P0 Architecture Risks

**⚠️ P0-1: `public/mascots/preview/` — 39 MB of unreferenced source PNGs live on production CDN**

These files are reachable at `https://pickupwords.pages.dev/mascots/preview/*.png`, each 1.3–1.7 MB. They serve no end-user purpose (0 source code references). They inflate every `wrangler pages deploy` by 39 MB. The service worker pattern `/mascots/*` will attempt to fetch them if any URL is visited. Fix: `mv public/mascots/preview tools/raw-mascots/preview`. No source changes. Zero risk.

**⚠️ P0-2: `fetchpriority="high"` on coin-gold (26px HUD icon) delays actual LCP image**

The browser's top-priority download slot is spent on a 178 KB coin icon rendered at 26×26 px after React mount — not the 88×auto `iso-grandma.webp` that is the largest above-fold element. On 3G this could shift LCP by 400–800 ms, pushing first-visit LCP over the 3000 ms R12 budget. Swap `fetchpriority` from `coin-gold.webp` to `iso-grandma.webp` in `index.html`. 1-line fix.

**⚠️ P0-3: `todo-永久token設定.md` is live on production at a guessable URL**

`https://pickupwords.pages.dev/todo-永久token設定.md` (reachable via URL-encoded form) exposes:
- All 6 of the author's repo names and their secret management approach
- The Infisical org/project structure
- Instructions to create Machine Identity tokens with TTL=0 (permanent)
- A placeholder `Client Secret: xxxxxxxxxxxxxxxxxxxxxxxxxxxx`

While no live token is in the file, the topology disclosure could accelerate an attacker who already has partial access. This file should never have been in `public/`. Move immediately to `docs/` or `tools/`.
